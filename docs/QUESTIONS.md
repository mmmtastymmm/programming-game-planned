## Open

- **Q58 (Language — env visibility to enemies).** Are a foreign bot's env values readable for free (like XP decals — you can see an enemy hauler runs `hurt_line=70`), gated behind that color's decryption %, or fully private? Free reading rewards attention and fits pillar 2; but `hurt_line` is exactly the number an attacker wants (burst it past the line before the handler fires), so free env may make custom thresholds self-defeating.
- **Q59 (Language — user-defined env keys).** V1 env keys are engine-defined only. Arbitrary user keys would be genuine bot-local *persistent memory* (state surviving restarts — which the variables-clear-on-restart rule deliberately forbids). Powerful and dangerous: enables stateful veterans and colony protocols, weakens the "re-derive your state" discipline, and gives fault loops a way to persist corrupted state. Never? Late unlock? Capped key count as a hardware stat (Memory bank)?
- **Q61 (Language — module semantics & limits).** Modules ([01-language.md](01-language.md)) formalize the Tier-4 program library. Open: unlock placement (bundled with `def`, or a separate construct right after it?); do imported lines count against per-bot **program memory** (a fat library raises every importing color's hardware bar — Q52's now-answered mechanism: over-bar members drop to the remainder) or is the closure exempt and only caps on module count/size apply?; can allies be granted modules the way they're granted channels (the co-op "shared library" scaffolding in [08-multiplayer.md](08-multiplayer.md) suggests yes)? *Update 2026-07-13:* import syntax + deploy-time resolution are DECIDED and live in the sandbox (Python's two forms; artifacts inline editor-generated `module <name>:` blocks; unknown module / unknown member / unimported qualified call / alias collisions are deploy errors; modules hold `def`s only for now — module `enum`s still to come). Still open besides the above: function-granularity tree-shaking (the sandbox ships whole imported modules).
- **Q71 (Terrain — how map generation actually works).** The docs state composition *goals* (center-out rings in [05-terrain.md](05-terrain.md); guaranteed start-zone Iron+Coal+Wood+Stone; Copper/Tin within first-expansion reach; Tin's sparsity floor; shorelines carrying Pumps and Sand Flats; nests farther out by arcanum; Crystal seeded near Corruption; PvP mirror-symmetric, co-op asymmetric with a shared frontier) — but no *procedure* produces them yet. Open: **algorithm family** — radial ring templates (the center-out guidance is already radial), noise-based biomes with constraint repair, hand-authored templates with procedural fill, or wave-function collapse? **Guarantees vs. emergence** — which properties are hard-validated post-gen (start-zone kit, path connectivity between every start and its required resources, no Bronze soft-lock) with regenerate-on-failure, and which are just weighted tendencies? **Symmetry** — reflective or rotational mirroring for PvP, and is it resource-*exact* (same vein amounts) or statistical? **Determinism** — mapgen runs in the sim from the match seed (same seeded-RNG discipline, map identity = part of the replay), or pre-generated and shipped as data with only its hash in the match setup? **Scaling** — map size and frontier depth per player count. Interacts with Q36 (mountain edges), Q38 (fords), Q63 (autotiling wants coherent regions), and biome cost overlays.
