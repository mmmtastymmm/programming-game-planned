## Open

- **Q35 (Terrain — Sand).** Sand as a heavy-cost tile (~5×, tuning constant). Open: does it need a distinguishing hook beyond cost, since Mud already owns "slow tile" (3×, 4× loaded)? Candidate: **sand punishes stopping** — idling more than N ticks on sand sinks the bot (escalating exit cost or a `stuck` fault), so `wait(n)` staging and rally points become unsafe in dunes. Mud punishes hauling routes; sand would punish loitering.
- **Q36 (Terrain — Mountain edge costs).** Should movement cost become a function of the **edge** (`from → to`) instead of the destination tile? Proposal: climbing onto a mountain is expensive, descending moderately expensive, ridge-to-ridge is plains-speed — mountain ranges become highways with costly on-ramps. Interacts with planned **High Ground** (ramp-gated, +2 sensor): do soft-climb mountains replace it (keeping the sensor bonus on summits) or coexist as the harder variant? Note: edge costs touch A*/`move_to`, so golden replay hashes change when this lands.
- **Q37 (Terrain — Ice).** Slide tile: entering ice continues the move in the same direction until non-ice (deterministic, 1×/tile, uncontrolled). Programs must plan slide endpoints; mass-produces `on bump:` use. Open: how do slides interact with bump freezes and one-way arrow overlays mid-slide?
- **Q38 (Terrain — Ford).** Shallow water: passable at high cost (~4×), possibly sensor-silent while wading. Makes bridges an upgrade rather than a hard gate — smooths the pre-terraforming game. Open: does this undercut the bridge/chokepoint economy that Water is designed to create?
- **Q39 (Terrain — Roads + integer cost scaling).** Terraformed road tile at half plains cost (the "tech tile" ground art). Requires scaling the base cost table ×2 (Plains 2, Road 1, Rubble 4, ...) since costs are integers — a one-time costs.ron migration that also buys finer tuning granularity for everything else. Decide before more cost-sensitive terrain ships.
- **Q40 (Terrain — wear & traces).** Two related "terrain remembers traffic" ideas: **Scree** that collapses after N crossings (per-tile counter, like natural bridge HP) so optimal programs rotate routes; and **footprints on sand** — decaying marks a future `tracks_at()` sensor could read, turning scouting from "where are they" into "where were they." Open: is per-tile mutable traffic state acceptable sim-size-wise, and is `tracks_at()` a Tier-2 sensor or later?
- **Q44 (Quirks — ship it? distribution).** Do quirks ([09-quirks.md](09-quirks.md)) ship at all, and behind what dial? Candidate: a match setting (Off / Rare / Standard), with Standard weighting most prints to 1 quirk and a minority to 0 or 2. Quirks change replay hashes, so they land behind the setting regardless.
- **Q45 (Quirks — acquisition mix).** Print-roll, scars-on-rescue, L5 track merits, corruption exposure: which subset makes v1? Print-roll alone is the cheapest to build; scars/merits do the most for pillar 3 but touch the rescue and XP systems.
- **Q46 (Quirks — removal/reroll).** Can a Repair Bay (or a pricier "Refit Bay") remove a scar or reroll a print quirk for resources? A cheap remove cheapens scars; no remove at all may make heavily-scarred veterans feel worse than reprinting, which fights the rescue incentive.
- **Q47 (Quirks — enemy visibility).** Are an enemy bot's quirks readable for free (visible wear/decals, like XP levels per pillar 2), gated behind that color's decryption %, or split (physical quirks visible, internal quirks like Memory Leak gated)?
- **Q48 (Quirks — introspection unlock).** `my_quirks()` / `has_quirk()`: free from the start (quirks are self-evident to their owner) or a cheap early function block? If quirks ship in a match where nobody can read them from code, they're pure noise — the builtin may need to be free whenever the setting is on.
- **Q52 (Agents — deploy-time stats vs. per-color programs).** Program memory and variable slots are per-bot hardware, but a color's source deploys to *every* bot of that color — what happens when a redeploy exceeds the weakest bot's memory? Candidates: reject the deploy listing under-spec bots; let under-spec bots keep the old version (version skew inside one color — probably unacceptable); auto-recall under-spec bots for scrap/re-coloring; a per-printer "minimum hardware" dial so its bots always fit its code. Whatever the answer, quirks stay out of deploy-time stats ([09-quirks.md](09-quirks.md) design rule 6).
- **Q53 (Agents — where does base sensor range live?).** The stat sheet ([02-agents.md](02-agents.md)) gives each chassis an innate sensor range (Scamp 8 … Drudge 5). Alternative: sight is a tool module ("Optics") competing for slots — blindness becomes a build choice, but every build feels taxed one slot. Or hybrid: innate chassis base, module extends. Interacts with High Ground (+2) and the Scouting track.
- **Q54 (Agents — signature/detectability stat).** Formalize "enemies sense this bot at ±N" (Loud Fans, Promiscuous Mode) as a **signature** stat: bot B is perceived by enemy E at `E.sensor_range + B.signature`. Requires asymmetric sensing in fog/vision code — perceiving and being perceived stop being the same radius. Worth the sim complexity, or should detectability effects be cut until a stealth system earns them?
- **Q55 (Agents — does the countdown explosion deal damage?).** With explosions now happening only at countdown expiry, is the blast cosmetic (the bot is simply gone) or an area-damage event? Area damage makes deliberate scuttles a weapon (the Hanged Man's ticking-wreck doorstep bombs, [04-enemies.md](04-enemies.md)) and makes clearing enemy wrecks urgent — but it also punishes standing near your *own* dying bots and adds a griefing vector on non-PvP servers (wreck-walk bombing). If yes: damage/radius scale with chassis? with cargo?
- **Q56 (Stats — integer granularity vs. percent modifiers).** Floor-and-clamp integer math makes percent effects dead on small bases: brownout's −50% can't touch a stock 1-cycle bot (0.5 → floor 0 → clamp 1), so the core over-extension penalty *only punishes upgraded CPUs*; Hauling's +10% of a Scamp's 4 cargo is +0; Building's +10% of 1 progress/tick is +0. Same disease Q39 found in integer terrain costs. Candidates: store fine-grained units (cycle budget in centicycles — the VM's cycle-debt accumulator already handles fractional accrual; cargo/progress ×10, the millitile precedent) or convert small-base percent perks to flat values. Decide before stats are implemented; replay hashes change either way.
- **Q57 (Agents/Terrain — sensor range: reveal vs. query, per-kind bonuses).** Fog is eyes-only *tile* reveal, but Combat L3 grants "+1 sensor range vs enemies" and the Ore-acle merit "+2 for resource nodes" ([09-quirks.md](09-quirks.md)) — a per-kind bonus can't reveal a tile "only for enemies." Ruling needed: is sensor range simultaneously the fog-reveal radius and the query radius (`closest`/`exists`/`scan_*`)? Do per-kind bonuses extend queries only, and may a query return an entity standing on an unrevealed tile? Interacts with High Ground (+2), Q38's sensor-silent wading, and Q54's signature stat.
- **Q58 (Language — env visibility to enemies).** Are a foreign bot's env values readable for free (like XP decals — you can see an enemy hauler runs `hurt_line=70`), gated behind that color's decryption %, or fully private? Free reading rewards attention and fits pillar 2; but `hurt_line` is exactly the number an attacker wants (burst it past the line before the handler fires), so free env may make custom thresholds self-defeating.
- **Q59 (Language — user-defined env keys).** V1 env keys are engine-defined only. Arbitrary user keys would be genuine bot-local *persistent memory* (state surviving restarts — which the variables-clear-on-restart rule deliberately forbids). Powerful and dangerous: enables stateful veterans and colony protocols, weakens the "re-derive your state" discipline, and gives fault loops a way to persist corrupted state. Never? Late unlock? Capped key count as a hardware stat (Memory bank)?
- **Q60 (Quirks — policy quirks via env: temperament vs. compulsion).** Policy quirks are implemented against the env registry ([09-quirks.md](09-quirks.md)): a **temperament** shifts a key's default (one `setenv` overrides it — the quirk evaporates under a good dotfile) while a **compulsion** clamps the key's range (permanent — tune the fear, can't remove it). Which strength for which quirk? And does `setenv` past a quirk clamp **clip** quietly (proposed — one program stays valid on every bot; `getenv` reports the landing value) or **fault** (louder, but the same line faults on one bot and not its twin)? Note scars probably *want* to be compulsions — a temperament scar is weightless on any colony with boot-window dotfiles.
- **Q61 (Language — module semantics & limits).** Modules ([01-language.md](01-language.md)) formalize the Tier-4 program library. Open: unlock placement (bundled with `def`, or a separate construct right after it?); do imported lines count against per-bot **program memory** (a fat library bloats every closure — inherits Q52's weakest-bot deploy problem) or is the closure exempt and only caps on module count/size apply?; can allies be granted modules the way they're granted channels (the co-op "shared library" scaffolding in [08-multiplayer.md](08-multiplayer.md) suggests yes)?
- **Q63 (Rendering — adaptive terrain art / autotiling).** Water autotiles in v1: each water tile picks one of 16 textures from a NESW same-neighbor bitmask (bank overlays appear against land; Bridge counts as water so the river visibly flows under the planks; off-map counts as water so edge rivers run clean off the board). Pure view layer — no sim state, no replay exposure. Queued ideas: an 8-bit blob set (47 variants) for seamless inner corners; **dual-grid marching squares for grass** so patches read as rounded organic shapes; variants keyed on same-neighbor *count* (denser tufts deep inside a field); a deterministic per-position hash picking among equivalent variants so large fields don't look stamped; subtle flow lines along long straight river runs. Open: which technique grass gets (blob set vs dual-grid), and whether the dual-grid's half-tile-offset render quads fight the slab-per-tile scene setup.

## Answered log

- **Q1–Q3, Q25** (Language): blocking channels (2×2 send/broadcast × try variants); hurt threshold as unlock; log buffer = hardware stat; channel espionage legal but gated by **comm keys** + decrypted names.
- **Q4–Q5** (Agents): upkeep is a data-driven mix, v1 = Energy + Metal; wreck countdown scales with XP.
- **Q6–Q8** (Resources): regen is a per-node-type data flag, mostly off; tiered harvest tools (Crystal high-tier); ally aid via Request Box, voluntarily fulfilled by hauling.
- **Q9–Q11** (Enemies): max arcanum is a match option, higher nests spawn farther out; mutations stay functional; Ferals reclaim nests.
- **Q12–Q14, Q29** (Terrain): terraforming in scope (build + deconstruct + cleanse); fog of war is eyes-only; Corruption radiates from sources; tall things block vision (Barricades are true walls, High Ground sees over).
- **Q15–Q18, Q28** (Progression): progression fully per-player; ALL function blocks learned at Template Caches (non-consumable schools, contest is territorial); any PvE earns constructs; prereqs per-player.
- **Q19–Q21, Q26** (Multiplayer): unanimous sim-speed votes with configurable cooldown; disconnected colonies run autonomously + decommission vote; indirect aggression always legal; PvP disconnect = free farm until reconnect.
- **Q22–Q24** (Architecture): `bevy_ecs` with sorted queries (rule in CLAUDE.md); A\* first, flow-fields note; byte-exact plain-text programs, version = source hash.
- **Q27** (Data): Data is a **currency** — the Research Archive runs a Data Exchange (Data → Chips/Metal, tuned rates). Never goes dead for veterans or in PvP.
- **Q30**: players **can hijack other players' wrecks** — XP-intact veteran theft is the third arm of the wreck race.
- **Q31 (revised)**: **Feral code is NOT open** — it decrypts by the same salvage attrition as player code, at per-arcanum rates (Fool leaks in ~2 kills; high arcana stay cryptic). Feral channels additionally require the nest's comm key via `analyze()`. One universal rule: programs are read on murder.
- **Q32**: full code disclosure **only after a match completes** (replays unlock post-match); live spectators see per-faction views.
- **Q33**: hijacked units are **never reprintable** — unique prizes.
- **Q34**: grace window stays in **ticks** — faster CPUs earning richer error handling is an intended hardware benefit. *(Later mooted: the 2026-07-13 reserved-handler redesign replaced the grace window with per-signal instruction caps.)*
- **Q41**: custom hurt threshold — **re-answered twice on 2026-07-13**: first restored as an `on hurt(n):` research, then superseded the same day by the **environment**: the threshold is the `hurt_line` env variable, set via `setenv` (the `F_ENV` cache unlock in [06-progression.md](06-progression.md)); runtime-adjustable, idiomatically configured in the `on boot:` window ([01-language.md](01-language.md) The Environment).
- **Q42–Q43**: **superseded by the 2026-07-13 reserved-handler redesign** — per-signal windows with instruction caps replace the shared grace clock (hurt is risk-priced again, Q42), and separate `on <signal>:` blocks remove the unified handler's `match` dependency (Q43).
- **Q49**: **signal-safety is one global `signal_safe` flag per function block**, not a per-signal matrix — assigned in 01's function table, stored in the registry as data (tunable like costs). First-pass calls: retreat verbs, sensors, logging, `try_*` sends, and `become_disabled` are safe; work, combat (`attack` — handlers are recovery, not fighting), and blocking calls are not.
- **Q51**: **window-reachable code is straight-line + `if`, all the way down** — user `def`s derive signal-safety (safe iff all calls safe, no loops, no recursion), each safe def gets a deploy-computed worst-case instruction count, and window calls charge that worst case against the cap. No runtime budget needed; boundedness is a compile-time property.
- **Q50**: **mooted by the abort redesign** — a second signal during any template (factory contents included) forces `abort`: logs go home, the bot wrecks, the rescue race starts. No humble carve-out needed; nothing is instantly erased, so the crash-loop-bumped-newbie case just wrecks rescueably.
- **Q62**: **what deploys is what leaks** — a color's decryption surface is its deployed, tree-shaken artifact (program + handler windows + transitively-called module functions). Never-deployed code never leaks. Shared module text sits under every deploying color's reveal mask and cross-reads across closures exactly like cross-version reading — shared libraries leak at the sum of their exposures; private forks are opsec. No new sim state ([01-language.md](01-language.md) Modules, [08-multiplayer.md](08-multiplayer.md)).
